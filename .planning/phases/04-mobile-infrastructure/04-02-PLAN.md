<task type="auto">
  <name>Task 1: Create EAS Build configuration</name>
  <files>apps/mobile/eas.json</files>
  <action>
Create `apps/mobile/eas.json` with three build profiles:

```json
{
  "cli": {
    "version": ">= 15.0.0",
    "appVersionSource": "remote"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": {
        "simulator": true
      },
      "android": {
        "buildType": "apk"
      }
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview"
    },
    "production": {
      "channel": "production",
      "autoIncrement": true
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "",
        "ascAppId": "",
        "appleTeamId": ""
      },
      "android": {
        "serviceAccountKeyPath": ""
      }
    }
  }
}
```

Notes on the profiles:
- **development**: For local simulator testing with dev client. iOS builds for simulator only, Android builds APK for easy sideloading.
- **preview**: For internal testing distribution (TestFlight/Google Play Internal). Uses "preview" update channel.
- **production**: For store submission. Uses "production" update channel with auto-incrementing build numbers.
- **submit**: Left with empty strings as placeholders -- these require Apple/Google account credentials that the user must configure in Phase 5 (MOBIL-02).

The `appVersionSource: "remote"` tells EAS to manage version numbers server-side, avoiding manual bumps.

- JSON is valid: `cd apps/mobile && node -e "JSON.parse(require('fs').readFileSync('eas.json','utf8')); console.log('Valid JSON')"`
  </verify>
  <done>eas.json exists with development (simulator), preview (internal), and production (store) build profiles</done>
