1. In `profiles.router.ts`, modify the `getProfileByUserLogged` procedure to read the `dev_impersonation` cookie from request headers BEFORE the existing profile lookup:

   ```typescript
   // At the top of the procedure handler:
   const cookieHeader = ctx.headers.get("cookie") ?? ""
   const impersonationMatch = cookieHeader.match(/dev_impersonation=([^;]+)/)

   if (impersonationMatch) {
     const cookieValue = decodeURIComponent(impersonationMatch[1] ?? "")
     const [realUserId, impersonatedProfileId] = cookieValue.split(":")

     if (realUserId === ctx.user.id && impersonatedProfileId) {
       const impersonatedProfile = await getProfileByProfileId(ctx.db, {
         id: impersonatedProfileId,
       })
       if (impersonatedProfile) return impersonatedProfile
     }
   }
   // ... existing getProfileByUserId call follows as fallback
   ```

   Parse from `ctx.headers.get("cookie")` — do NOT use `cookies()` from `next/headers` (unreliable in tRPC context per research). Validate that `realUserId === ctx.user.id` to prevent cross-user impersonation. If the impersonated profile doesn't exist, fall through to real profile.

2. Add a new `getRealProfile` procedure (protectedProcedure, no input) that always returns the real user's profile by calling the existing `getProfileByUserId(ctx.db, { userId: ctx.user.id })` directly — ignoring any impersonation cookie. This is needed by the Developer Tools page to get the `realUserId` for cookie construction.

3. Import `getProfileByProfileId` from the funcs file at the top of the router.
